## 🚀 Быстрый старт

### Первое устройство (инициализация)

```bash
# 1. Настройка конфигурации
cd database/
cp .env.example .env          # Создать файл конфигурации
# Откройте .env и проверьте значения

# 2. Запуск базы данных
docker-compose up -d

# 3. Инициализация схемы (если БД пустая)
docker cp BD_dead_kolesa_kz.sql car_parts_db:/tmp/setup.sql
docker exec -it car_parts_db psql -U myuser -d parts_catalog -f /tmp/setup.sql

# 4. Сделать дамп и запушить
chmod +x sync_push.sh sync_pull.sh sync_status.sh apply_migrations.sh
./sync_push.sh "initial: первый дамп БД"
```

### Остальные участники (получение)

```bash
# 1. Клонировать репозиторий
git clone <URL> Projects_CA
cd Projects_CA/database/

# 2. Настроить конфигурацию
cp .env.example .env

# 3. Получить дамп и восстановить БД
chmod +x sync_push.sh sync_pull.sh sync_status.sh apply_migrations.sh
./sync_pull.sh
```

## 👥 Рабочий процесс команды

### Цикл разработки (4 человека)

```
┌─────────────────────────────────────────────────────────────┐
│  Участник 1: Разрабатывает фичу → меняет БД                 │
│       │                                                       │
│       ▼                                                       │
│  ./sync_push.sh "добавлено поле warranty_days"              │
│       │                                                       │
│       ▼                                                       │
│  ───────────────── GitHub ──────────────────                 │
│       │                                                       │
│       ▼                                                       │
│  Участники 2,3,4:                                            │
│       ./sync_pull.sh   ← получают дамп + миграции            │
│       ./sync_status.sh ← проверяют что всё OK                │
└─────────────────────────────────────────────────────────────┘
```

### Правило: **Один работает → остальные получают**

> ⚠️ **Важно:** Только **один человек** вносит изменения в БД за раз.
> Остальные получают изменения через `sync_pull.sh`.

### Типичный цикл:

| Шаг | Действие | Кто |
|-----|----------|-----|
| 1 | Делает фичу, меняет БД | Участник 1 |
| 2 | `./sync_push.sh "описание"` | Участник 1 |
| 3 | `./sync_pull.sh` | Участники 2, 3, 4 |
| 4 | `./sync_status.sh` — проверка | Все |
| 5 | Проверили — всё работает ✓ | Все |

## 🛠 Скрипты и их назначение

### `sync_push.sh` — Отправить изменения

```bash
./sync_push.sh ["Описание изменений"]
```

**Что делает:**
1. ✅ Загружает конфиг из `.env`
2. ✅ Проверяет Docker и PostgreSQL
3. ✅ **Делает `git pull`** — проверяет, нет ли новых изменений на GitHub
4. ⚠️ Если есть обновления — **спрашивает подтверждение** (защита от перезаписи)
5. ✅ Создаёт дамп БД с **валидацией** (проверка кодировки, содержимого)
6. ✅ Обновляет `db_manifest.json` (версия, кол-во записей, хэш)
7. ✅ Коммитит и пушит

**Защита от конфликтов:** Если на GitHub есть более новая версия — скрипт предупредит и спросит подтверждение.

---

### `sync_pull.sh` — Получить изменения

```bash
./sync_pull.sh
```

**Что делает:**
1. ✅ Делает `git pull`
2. ✅ **Валидирует дамп** перед восстановлением:
   - Проверка что файл не пустой
   - Проверка что это SQL
   - Проверка кодировки UTF-8
   - Проверка что таблица `parts_inventory` есть
3. ✅ **Создаёт резервную копию** текущей БД (на случай отката)
4. ✅ Восстанавливает БД из дампа
5. ✅ **Применяет pending миграции** (новые ALTER TABLE и т.д.)
6. ✅ Проверяет целостность восстановленной БД
7. ✅ **Автоматический откат** если дамп битый

**Защита от ошибок:** Если дамп поврежён — БД будет откачена к предыдущей версии.

---

### `sync_status.sh` — Проверить состояние

```bash
./sync_status.sh
```

**Показывает:**
- 🟢 Статус контейнера
- 📊 Версию схемы и количество записей
- 🕐 Последнюю синхронизацию (когда, с какого устройства)
- 📦 Статус миграций (применены / ожидают)
- 🔍 Git-статус (есть ли локальные изменения, расхождения с remote)

**Рекомендуется запускать после `sync_pull.sh`** для проверки.

---

### `apply_migrations.sh` — Только миграции (без потери данных)

```bash
./apply_migrations.sh
```

**Когда использовать:**
- Кто-то добавил новую миграцию в `migrations/`
- Вы сделали `git pull` и получили новый `.sql` файл
- **НЕ нужно** пересоздавать всю БД — только применить изменения схемы
- Данные в БД **сохраняются**

## 📁 Структура файлов

```
database/
├── .env.example              # Шаблон конфигурации (коммитится)
├── .env                      # Локальная конфигурация (НЕ коммитится!)
├── db_manifest.json          # Манифест: версия схемы, записи, хэш
│
├── sync_push.sh              # Отправить изменения
├── sync_pull.sh              # Получить изменения
├── sync_status.sh            # Проверить состояние
├── apply_migrations.sh       # Только миграции
│
├── full_backup.sql           # Актуальный дамп БД (коммитится)
├── BD_dead_kolesa_kz.sql     # Схема БД (для первого запуска)
├── docker-compose.yml        # PostgreSQL контейнер (использует .env)
│
└── migrations/               # SQL-миграции для изменения схемы
    └── 001_add_shop_url_column.sql
```

## 🔧 Решение проблем

### ❌ "Контейнер не запущен"

```bash
cd database/
docker-compose up -d
# Подождать 10-15 секунд
./sync_status.sh
```

### ❌ "Дамп не содержит таблицу parts_inventory"

Это значит что дамп от **старой версии схемы** или битый.

```bash
# Проверить содержимое дампа
head -20 database/full_backup.sql

# Если дамп битый — попросите кого-то заново запушить:
# Участник с рабочей БД: ./sync_push.sh "fix: обновить дамп"
# Вы: ./sync_pull.sh
```

### ❌ Ошибка UTF-8 / кодировки

```bash
# Проверить кодировку дампа
file database/full_backup.sql

# Если кодировка не UTF-8 — пересохраните:
iconv -f WINDOWS-1251 -t UTF-8 database/full_backup.sql > database/full_backup_utf8.sql
mv database/full_backup_utf8.sql database/full_backup.sql
git add database/full_backup.sql
git commit -m "fix: исправить кодировку дампа"
git push
```

### ❌ Конфликт при git pull

```bash
# Вариант 1: Взять удалённую версию (рекомендуется)
git checkout -- database/full_backup.sql
git pull
./sync_pull.sh

# Вариант 2: Сохранить локальную, потом смержить
git stash
git pull
git stash pop
# Разрешить конфликт вручную
./sync_push.sh
```

### ❌ Миграция не применяется (column already exists)

```bash
# Это нормально — миграция уже применена
# Проверить статус:
./sync_status.sh

# Если нужно пропустить — добавить в db_manifest.json:
python3 -c "
import json
with open('database/db_manifest.json', 'r') as f:
    m = json.load(f)
m.setdefault('migrations_applied', []).append('002_new_column.sql')
with open('database/db_manifest.json', 'w') as f:
    json.dump(m, f, indent=2)
"
```

### ❌ "Расхождение! N локальных и M удалённых коммитов"

```bash
# Выполните sync_status.sh для подробностей
./sync_status.sh

# Если есть локальные неотправленные:
./sync_push.sh

# Если есть удалённые неполученные:
./sync_pull.sh

# Если и там и там (расхождение):
git pull --rebase
./sync_push.sh
```

### ❌ БД повреждена — нужен полный сброс

```bash
cd database/

# 1. Удалить контейнер и volume
docker-compose down -v

# 2. Пересоздать
docker-compose up -d

# 3. Восстановить из дампа
./sync_pull.sh
```

## 📝 Как добавить миграцию

Когда нужно **изменить структуру БД** (добавить колонку, индекс и т.д.):

### 1. Создать файл миграции

```bash
# Формат имени: NNN_описание.sql
touch database/migrations/002_add_warranty_column.sql
```

### 2. Написать SQL

```sql
-- database/migrations/002_add_warranty_column.sql
-- Миграция: добавить поле warranty_days

ALTER TABLE parts_inventory
ADD COLUMN IF NOT EXISTS warranty_days INTEGER DEFAULT 0;

CREATE INDEX IF NOT EXISTS idx_warranty ON parts_inventory(warranty_days);
```

> ⚠️ **Всегда используйте `IF NOT EXISTS`** — миграция может применяться повторно.

### 3. Запушить изменения

```bash
# Участник 1 (кто создал миграцию):
./sync_push.sh "feat: добавить поле warranty_days"
```

### 4. Остальные участники

```bash
# Автоматически применится при pull:
./sync_pull.sh

# Или вручную (если данные терять нельзя):
./apply_migrations.sh
```

### 5. Проверить

```bash
./sync_status.sh
# В секции "Миграции" должно быть:
#   ✓ 001_add_shop_url_column.sql (применена)
#   ✓ 002_add_warranty_column.sql (применена)
```

## 🔐 Безопасность

- **`.env` файл НЕ коммитится** в Git — каждый участник создаёт свой
- Пароли хранятся только локально
- Для приватного репозитория GitHub — допустимо использовать одинаковые пароли в `.env`
- Для публичного репозитория — **обязательно** используйте уникальные пароли

## 📊 Как это работает (архитектура)

```
┌─────────────────────────────────────────────────────────┐
│                    GitHub Repository                      │
│  full_backup.sql  +  db_manifest.json  +  migrations/   │
└──────────────────────┬──────────────────────────────────┘
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
     Участник 1   Участник 2   Участник 3
     (push)       (pull)       (pull)
          │            │            │
          ▼            ▼            ▼
     ┌─────────────────────────────────┐
     │  local PostgreSQL (Docker)      │
     │  parts_inventory: 247 записей   │
     └─────────────────────────────────┘
```

**Принцип:** Один дамп = один источник правды.
**Защита:** pull-before-push, валидация, откат при ошибке.
